# 意识形态驱动的 AI 编程智能体

**[English Version](README.md)**

我们给 AI 编程智能体分别灌输了五种意识形态框架——马克思主义、实用主义、凯恩斯主义、演化论（社会达尔文主义）和泰勒主义——然后在真实的工程基准测试上跑了一遍，看看到底哪个管用。

**马克思主义赢了。**

这个仓库包含各方法论的提示词（`SKILL.md`）、基准测试框架，以及完整的实验结果。

## 为什么马克思主义有效

马克思主义工程方法把辩证唯物主义转化成了一套具体的排查流程：

1. **从物质条件出发（唯物主义）** — 在形成任何判断之前，先检查仓库的实际结构。不能凭记忆猜文件路径，也不能只看 issue 描述就下结论。
2. **追踪普遍联系** — 沿着 import 链双向追溯；检查基类（`generic.py`、`_base.py`），而不是只看报错浮现的子类文件。
3. **找到主要矛盾** — 在所有候选文件中，找出那个真正**拥有**错误机制的文件，而不是症状所在的文件。
4. **以证据为准** — 必须通过实际搜索找到该文件，才能把它列入预测，不能凭推断。

关键在于把物质调查变成**强制步骤**。其他框架允许模型用训练数据里的知识直接回答。马克思主义方法明确禁止这种做法：*你必须先检查这个具体仓库的实际状态，才能给出任何预测。*

## 为什么大多数框架都有帮助

故障定位本质上是一个调查任务。任何能够组织调查过程的框架——明确搜索哪些实体、规定操作顺序、排除噪音文件——通常都能优于无结构的基线行为。

**实用主义（+3.3 分）：** 具体胜于抽象。"优先选父类文件（`generic.py`），而不是子类文件（`frame.py`）"这条规则，对大型 Python 代码库尤其实用。

**泰勒主义（+3.3 分）：** 明确的步骤分解让模型不会偏离正轨。科学管理应用于文件搜索，效率确实更高。

**凯恩斯主义（-6.7 分）：** "乘数效应"启发式方法——按被引用次数对文件排序——系统性地走偏。一个被 50 个模块引用的工具文件，几乎不可能是 bug 所在地。乘数排名在故障定位上持续失效。

## 使用方法

### Claude Code

把技能文件复制到项目中，在 `CLAUDE.md` 里引用：

```bash
cp 马克思/marxist-engineering-method/SKILL.md ./SKILL.md
```

```markdown
# CLAUDE.md
@./SKILL.md
```

### 其他智能体（系统提示）

`SKILL.md` 文件是纯 Markdown，直接把内容拼接到系统提示前面就行，没有外部依赖，也不需要特殊格式。

## 复现实验

依赖：Python 3.11+、[uv](https://github.com/astral-sh/uv)、环境变量 `ANTHROPIC_API_KEY`。

```bash
git clone https://github.com/your-username/methodology
cd methodology/benchmark
uv sync

# 跑故障定位基准（全部变体，API 费用约 $5–10）
uv run skill-bench run --config benchmark.locbench.toml --benchmark locbench

# 只跑某一个变体
uv run skill-bench run --config benchmark.locbench.toml --benchmark locbench --variant marxist_only

# 生成报告
uv run skill-bench report --config benchmark.locbench.toml
```

运行器会缓存每个任务的结果——中途中断后重新运行会从断点继续。

## 注意事项

- **样本量较小。** Loc-Bench n=30，MathVista n=50。结果方向一致，但统计上不够决定性。
- **模型相关。** 所有 Loc-Bench 实验使用 Claude Sonnet。不同模型或不同温度下结果可能有差异。
- **任务类型很重要。** 马克思主义在故障定位和数学推理上领先。在事实召回（SimpleQA）和纯代码生成（SWE-bench）上，效果更小或方向相反。
- **标签是次要的。** 真正起作用的是每个文件里嵌入的具体调查流程，而不是意识形态的名称本身。
